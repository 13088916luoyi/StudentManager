# EARS格式指南

## 概述
EARS (Easy Approach to Requirements Syntax) 是一种简化的需求语法方法，用于编写清晰、可测试的需求。

## EARS模式

### 1. 普遍性需求 (Ubiquitous)
**格式**: `系统应当[功能]`
**适用**: 系统必须始终满足的需求
**示例**: `系统应当验证用户身份`

### 2. 事件驱动需求 (Event-driven)
**格式**: `当[触发条件]时，系统应当[功能]`
**适用**: 由特定事件触发的需求
**示例**: `当用户点击登录按钮时，系统应当验证用户名和密码`

### 3. 状态驱动需求 (State-driven)
**格式**: `当[状态]时，系统应当[功能]`
**适用**: 系统处于特定状态时的需求
**示例**: `当系统处于维护模式时，系统应当拒绝所有用户登录`

### 4. 可选功能需求 (Optional)
**格式**: `当[功能]被启用时，系统应当[行为]`
**适用**: 可选功能的需求
**示例**: `当自动保存功能被启用时，系统应当每5分钟自动保存用户数据`

### 5. 异常处理需求 (Exception)
**格式**: `当[异常情况]时，系统应当[处理行为]`
**适用**: 错误和异常处理
**示例**: `当数据库连接失败时，系统应当显示错误提示并记录日志`

## 编写原则

1. **可测试性**: 每个需求必须可验证
2. **无歧义性**: 使用清晰、明确的语言
3. **完整性**: 包含所有必要信息
4. **一致性**: 避免矛盾的需求
5. **原子性**: 每个需求只描述一个功能

## 验收标准格式

每个需求应包含：
- **Given (前置条件)**: 系统初始状态
- **When (触发动作)**: 用户操作或系统事件
- **Then (预期结果)**: 系统应达到的状态

示例：
```
Given: 用户在登录页面
When: 用户输入正确的用户名和密码并点击登录
Then: 系统应当跳转到用户主页
```
